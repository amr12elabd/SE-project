**Project:** PopEyez — Pop-Up Café Event Management Platform  
**Course:** Software Engineering (University Project)

---

## How AI Was Used

### Phase 1: Project Architecture & Planning

**Prompt summary:**
> "Design the database schema for a pop-up café event management platform with 5 user roles: Organizer, Staff, Vendor, Guest, and Venue Owner. The platform should handle event lifecycle from planning to post-event reporting."

**AI contributed:**
- Recommended 15 Mongoose models with appropriate field types, refs, and enums
- Suggested pre-save hooks for password hashing and sentiment auto-classification
- Proposed JWT + bcrypt for auth with role-based middleware pattern

**Human decisions:**
- Chose to keep roles as a single `role` field on User (vs. separate role collections)
- Decided on EGP as sole currency for Cairo-specific context
- Scoped the project to local development (no cloud deployment)

---

### Phase 2: Backend Development

**Prompts included:**
- "Generate the Express.js server setup with CORS for localhost:5173"
- "Write the auth middleware that verifies JWT and attaches user to request"
- "Create the Event controller with role-based filtering"

**AI contributed:**
- All 17 Express route files with proper middleware chains
- 17 controllers with CRUD operations, role filtering, and nested resource handling
- Error handler middleware covering Mongoose validation, cast errors, and duplicates
- `mergeParams: true` pattern for nested event sub-routes (budget, layout, feedback, etc.)

**Human review and changes:**
- Added the `getEventDashboard` aggregation endpoint after realizing the organizer dashboard needed combined stats
- Simplified the `Communications` model to use a flat `seenBy` array instead of sub-documents
- Decided to store `qrCodeValue` as UUID in the Guest model rather than generating QR on-demand

---

### Phase 6: Bug Fixes & Integration

**Issues found and fixed with AI assistance:**

1. **API method name mismatches** — Some pages used `vendorsAPI.getMyProfile()` while the API file defined `getProfile()`. Fixed by adding aliases in `api/index.js`.

2. **Guest invitations endpoint** — The `getGuestInvitations` controller originally required an `email` query parameter. Updated it to fall back to `req.user.email` for authenticated guests.

3. **Venue report endpoint missing** — The `VenueReports.jsx` page called `reportsAPI.getVenueReport(venueId)` but no such endpoint existed. Added `GET /api/venues/:id/report` controller and route.

4. **Communications markSeen mismatch** — The `DayOfMessages.jsx` page initially called `commsAPI.markSeen(eventId)` with wrong signature. Removed the call since guests don't need to mark individual messages; the backend tracks `seenBy` per message via the organizer's follow-up tool.

## AI Limitations Encountered

1. **Context window constraints** — The full project exceeded one conversation's context window. The session was continued across multiple turns with summary context.

2. **Hallucinated method names** — AI sometimes generated frontend code calling API methods that hadn't been defined yet (e.g., `getOwnerVenues` vs `getMyVenues`). These were caught during integration review.

3. **No runtime testing** — AI cannot run the code. All generated code required manual testing against the running application.

---

## Human Contributions & Review

Despite AI code generation, significant human judgment was applied:

- **Architecture decisions**: Chose single-role users, Cairo-specific context, no cloud dependencies
- **Scope management**: Decided which features to simplify (layout editor, availability calendar, messaging)
- **Integration review**: Caught API signature mismatches between frontend and backend
- **Content review**: Verified that seed data was realistic and consistent across collections
- **Documentation**: Authored the assumptions.md and schema-notes.md content

## Academic Integrity Statement

This project used AI as a code generation and architectural guidance tool, similar to how developers use GitHub Copilot or Stack Overflow. All code was reviewed, understood, and adapted by the project author. The AI did not make design decisions independently — all major choices (technology stack, data models, user flows) were directed by the project author based on the Software Engineering course requirements.
